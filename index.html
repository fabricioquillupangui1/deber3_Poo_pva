Programaci√≥n Orientada a Objetos (POO)

Conceptos Fundamentales: Clases, Objetos y los 4 Pilares

üß† Proceso de Pensamiento para Modelar con Objetos

La POO nos permite construir software modelando el mundo real. Este proceso l√≥gico nos gu√≠a para transformar un problema en una soluci√≥n estructurada y modular.

Pasos Clave para el Modelado

Paso Concepto Ejemplo (Entidad: Estudiante)
1. Identificaci√≥n de Entidades Determinar las "cosas" (Clases) importantes en el problema. Identificar la clase `Estudiante`.
2. Definici√≥n de Atributos Describir las caracter√≠sticas (datos) que definen a la entidad. Atributos: `nombre`, `edad`, `carrera`.
3. Definici√≥n de Comportamiento Establecer las acciones (M√©todos) que puede realizar la entidad. M√©todos: `inscribirse()`, `presentarExamen()`.
4. Establecimiento de Relaciones Definir c√≥mo interact√∫an las clases (Herencia, Composici√≥n, etc.). Relaci√≥n: `Estudiante` se relaciona con `Curso`.

üß± Clases y Objetos

Clase: El Plano

Una Clase es la definici√≥n abstracta. Act√∫a como una plantilla o molde que especifica la estructura (atributos) y el comportamiento (m√©todos) que tendr√°n los objetos creados a partir de ella. No consume memoria hasta que se instancia.

Objeto: La Instancia

Un Objeto es una instancia concreta de una clase. Es una entidad f√≠sica en la memoria que ha sido construida siguiendo las especificaciones de la clase, y tiene valores espec√≠ficos para sus atributos.

Analog√≠a: La clase `MoldeDeGalletas` (la plantilla) se usa para crear el objeto `miGalleta` (la galleta horneada y espec√≠fica).

üõ°Ô∏è Los 4 Pilares Fundamentales de la POO

Estos principios son la base te√≥rica para el dise√±o de software robusto y mantenible.

1. Encapsulamiento

Agrupa datos y m√©todos en una unidad y restringe el acceso directo a los datos internos, permitiendo la interacci√≥n solo a trav√©s de m√©todos p√∫blicos. Protege los datos.

(Datos seguros dentro de la "caja fuerte")

2. Abstracci√≥n

Muestra solo la funcionalidad esencial al usuario y oculta los detalles de implementaci√≥n complejos. Permite interactuar con objetos a un alto nivel. Simplifica el uso.

(Usar un televisor sin saber c√≥mo funciona)

3. Herencia

Permite que una nueva clase (subclase) reutilice, extienda o modifique atributos y m√©todos de una clase ya existente (superclase). Establece una relaci√≥n "Es Un". Promueve la reutilizaci√≥n.

(Clase Padre pasa propiedades a Clase Hija)

4. Polimorfismo

Habilidad de un m√©todo para tener diferentes implementaciones dependiendo de la clase que lo ejecute. Significa "muchas formas". Permite la flexibilidad.

(El mismo m√©todo 'dibujar()' se comporta diferente)

Ejemplo 1: Sistema de Gesti√≥n de Biblioteca (Herencia y Encapsulamiento)

Modelamos √≠tems de la biblioteca, usando la clase base abstracta ItemBiblioteca para definir el comportamiento com√∫n y aplicando (Encapsulamiento) en el estado de disponibilidad.


// 1. Clase Base Abstracta (Abstracci√≥n)
abstract class ItemBiblioteca(val titulo: String, val autor: String) {
    // ENCAPSULAMIENTO: Atributo privado que guarda el estado real
    private var _disponible: Boolean = true

    val isDisponible: Boolean
        get() = _disponible // Getter p√∫blico de solo lectura

    // ABSTRACCI√ìN: M√©todos de interfaz para interactuar con el estado interno
    fun prestar() {
        if (_disponible) {
            _disponible = false
            println("$titulo ha sido prestado.")
        } else {
            println("$titulo NO est√° disponible para pr√©stamo.")
        }
    }
    
    abstract fun mostrarDetalles()
}

// 2. Clase Hija (Herencia)
class Libro(
    titulo: String, 
    autor: String, 
    val isbn: String
) : ItemBiblioteca(titulo, autor) { // HERENCIA
    
    override fun mostrarDetalles() {
        println("Libro: $titulo, ISBN: $isbn")
        println("Estado: " + if (isDisponible) "Disponible" else "Prestado")
    }
}

// 3. Uso
fun main() {
    val libro1 = Libro("El Quijote", "Cervantes", "978-XXXX")
    libro1.mostrarDetalles()
    libro1.prestar() // Modifica el estado interno (Encapsulamiento)
    libro1.mostrarDetalles()
}
Ejemplo 2: C√°lculo de √Åreas de Figuras Geom√©tricas (Polimorfismo y Abstracci√≥n)

Implementamos la interfaz Figura (Abstracci√≥n) y forzamos a que cada clase hija implemente el m√©todo calcularArea() de manera diferente, demostrando el (Polimorfismo).


// 1. Interfaz (Abstracci√≥n: Contrato)
interface Figura {
    // POLIMORFISMO: Define el m√©todo que tendr√° m√∫ltiples formas
    fun calcularArea(): Double
    fun dibujar()
}

// 2. Clase Circulo (Herencia e implementaci√≥n polim√≥rfica)
class Circulo(val radio: Double) : Figura {
    
    // POLIMORFISMO: Forma 1 de calcularArea()
    override fun calcularArea(): Double {
        return Math.PI * radio * radio 
    }
    
    override fun dibujar() {
        println("Dibujando un C√≠rculo.")
    }
}

// 3. Clase Rectangulo (Herencia e implementaci√≥n polim√≥rfica)
class Rectangulo(val ancho: Double, val alto: Double) : Figura {
    
    // POLIMORFISMO: Forma 2 de calcularArea()
    override fun calcularArea(): Double {
        return ancho * alto
    }
    
    override fun dibujar() {
        println("Dibujando un Rect√°ngulo.")
    }
}

// 4. Uso (Demostraci√≥n de Polimorfismo)
fun main() {
    val figuras = listOf(Circulo(5.0), Rectangulo(4.0, 6.0))

    for (figura in figuras) {
        figura.dibujar() 
        // Llama a la implementaci√≥n espec√≠fica de cada objeto (Polimorfismo)
        println("√Årea: " + figura.calcularArea()) 
    }
}

package com.ejemplo.bancario.logica 

// CLASE BASE: ABSTRACCI√ìN y ENCAPSULAMIENTO
abstract class CuentaBancaria(val titular: Cliente, protected var saldo: Double) {

    fun obtenerSaldo(): Double = saldo
    
    fun depositar(cantidad: Double): String {
        return if (cantidad > 0.0) {
            saldo += cantidad
            "Dep√≥sito exitoso. Nuevo saldo: $saldo"
        } else {
            "El dep√≥sito debe ser mayor a cero."
        }
    }

    // POLIMORFISMO
    abstract fun retirar(cantidad: Double): String
}

// CUENTA CORRIENTE
class CuentaCorriente(titular: Cliente, saldoInicial: Double) : CuentaBancaria(titular, saldoInicial) {
    
    override fun retirar(cantidad: Double): String {
        return if (cantidad <= saldo) {
            saldo -= cantidad
            "Retiro CC exitoso. Nuevo saldo: $saldo"
        } else {
            "ERROR: Fondos insuficientes en Cuenta Corriente."
        }
    }
}

// CUENTA AHORROS
class CuentaAhorros(titular: Cliente, saldoInicial: Double, val comision: Double = 2.0) : CuentaBancaria(titular, saldoInicial) {
    
    override fun retirar(cantidad: Double): String {
        val totalRetirar = cantidad + comision
        return if (totalRetirar <= saldo) {
            saldo -= totalRetirar
            "Retiro CA de $cantidad (Comisi√≥n: $comision). Nuevo saldo: $saldo"
        } else {
            "ERROR: Fondos insuficientes para cubrir el retiro y la comisi√≥n ($totalRetirar)."
        }
    }
}

    }

class EmpleadoPorHora(nombre: String, id: Int, val tarifaHora: Double, val horasTrabajadas: Double) : Empleado(nombre, id) {
    
    override fun calcularSalario(): Double {
        return tarifaHora * horasTrabajadas
    }
}
